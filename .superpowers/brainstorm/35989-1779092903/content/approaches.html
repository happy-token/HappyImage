三个重设计方向

核心问题：image-cards、infographic 等主力技能没有 main.ts，必须在 Claude Code 会话中以 /slash 命令调用，claude -p 根本没有走技能逻辑。

A

命令导出优先 — 做好参数配置器

放弃有问题的 in-browser 执行，专注把"浏览风格 → 填内容 → 生成命令"做到极致。用户复制命令到 Claude Code CLI 运行，质量和直接用 CLI 完全一致。

新增：设置页面管理 API Key / 偏好；向导压缩为 3 步；全面视觉重设计；覆盖所有 22 个技能。

优点

  • 结果质量和 CLI 完全一致
  • 最简单可靠，可快速交付
  • 不需要改执行架构

缺点

  • 用户仍需切换到终端执行
B

终端嵌入 — 浏览器里跑真实 Claude Code

用 xterm.js 在浏览器嵌入终端，后端通过 PTY (node-pty) 启动真实 Claude Code 会话，自动输入 /baoyu-xxx 命令。用户看到和 CLI 完全相同的交互过程。

新增：设置页 + 向导 + 终端面板三栏布局；API Key 配置；实时流式输出。

优点

  • 执行质量和 CLI 完全一致
  • 用户不离开浏览器
  • 可以看到完整 Claude 思考过程

缺点

  • 需要 node-pty + xterm.js，工程量大
  • 终端 UI 在浏览器里体验不一定好
C

SDK 直调 — 用 Anthropic API 重实现执行

完全绕过 Claude Code CLI，后端用 Anthropic SDK 直接调用 API，把 SKILL.md 作为 system prompt 注入，Web UI 接管 AskUserQuestion 的 Q&A 流程，变成向导步骤。

新增：需要自己实现技能的 Q&A 逻辑；streaming 流式展示结果；图片结果内嵌展示；无需安装 Claude Code CLI。

优点

  • 完全浏览器内，体验流畅
  • 不依赖本地 CLI
  • 可以做精美的流式 UI

缺点

  • 工程量最大，需要重实现技能逻辑
  • 结果可能和 CLI 仍有差距
  • 依赖 API Key 直接消费（不走 Claude Code）

我的推荐：A 方案，先把命令配置器做到极致，稳定交付。B 方案可作为后续 P1 追加。
